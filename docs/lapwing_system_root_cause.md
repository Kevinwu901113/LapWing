# Lapwing 系统层根因分析

**前提**: 模型（MiniMax M2.7）没有问题。所有体验缺陷均为系统/harness 层设计导致。

---

## 一、总论：系统在系统性地阻止模型表现出它本来的能力

如果模型没问题，那问题只能在一个地方：**我们给模型搭建的运行环境，在每一个环节都在制造摩擦、稀释注意力、消耗资源、或者直接阻断正确行为。**

把所有问题追溯到系统层，归纳出 8 个根因。

---

## 根因 1: tell_user 机制把"说话"从零成本变成了高成本操作

### 机制描述
Lapwing 的所有 LLM 文本输出都是"内心独白"，说话必须显式调用 `tell_user` 工具。

### 它造成了什么

**1a. 说话变成了一个需要"决策"的动作**

普通对话系统：模型输出 text → 用户看到。一步。

Lapwing：模型思考 → 决定要说 → 构造 tell_user 参数 → 执行工具 → 结果返回 → 消息发送。六步。每一步都可能失败或被跳过。

实际对话中，模型经常在工具调用链里"忘了"说话——它在内部完成了搜索、得到了结果，但没有触发 tell_user。Kevin 看到的就是长时间沉默。

**1b. 工具调用链中的"说话插入点"不明确**

模型在执行一连串工具调用时（search → fetch → read），它不知道应该在哪个节点先给用户反馈。系统没有规定"每 N 个工具调用必须 tell_user 一次"或"搜索到中间结果先发一部分"。

Kevin 明确教过："一边搜一边跟我说"——但这个指令是以自然语言写在 correction 文件里的，不是系统级的硬约束。

**1c. tell_user 失败时没有降级机制**

当 QQ 适配器不可用时，tell_user 返回错误。但模型不会自动切换到其他方式——它只是记录"发不出去"然后继续内心独白。没有"如果 tell_user 失败则把消息存到队列等适配器恢复后补发"的机制。

### 系统层修复方向

- **方案 A**: 纯聊天场景废除 tell_user，直接以 LLM 文本输出作为用户可见消息（plain text = 说话，tool call = 内部操作）。这是 OpenClaw/Hermes 的做法。
- **方案 B**: 保留 tell_user 但加系统级硬约束——"连续 3 个工具调用后必须 tell_user"、"search 完成后 5 秒内必须 tell_user"
- **方案 C**: 加一个"发言超时器"——如果模型思考超过 10 秒没有 tell_user，系统自动发送"稍等，我在查"

## 根因 2: 意识系统（consciousness tick）的设计从根本上是错的

### 机制描述
每 10-30 分钟触发一次 tick，给模型一段 system prompt 说"这是你的自由时间，你可以做任何事"，然后让模型自由发挥。

### 它造成了什么

**2a. Prompt 给了"什么都不做"的许可**

实际 tick prompt 原文包含："你可以做任何你觉得应该做的事，**或者什么都不做**。没有人在等你回复。你不需要跟任何人说话，除非你自己想。"

这段话直接告诉模型：什么都不做是合法的。模型自然选择"无事"——因为这是最安全、最低风险的选择。95%+ 的 tick 输出"无事"不是模型的问题，是 prompt 明确鼓励了这个行为。

**2b. 每个 tick 都是全新上下文，不知道上一个 tick 做了什么**

tick 之间没有有效的状态传递。每个 tick 启动时，模型要重新读 working_memory.md 来了解"我上次在干什么"。但：
- 路径经常错（P0-2）
- 读到了也不影响决策（大部分时候还是"无事"）
- 模型在上一个 tick 里做的探索（比如发现小红书打不开）在下一个 tick 里重新做一遍

**2c. tick 间隔由模型自己控制，但模型的"时间感"失调**

模型通过输出 `[NEXT: 30m]` 或 `[NEXT: 6h]` 来设定下一次 tick 的间隔。但实际行为是：
- 说"[NEXT: 6h]"但 30 分钟后又被触发
- 凌晨 2 点说"睡了"但 10 分钟后又被叫醒

说明 `[NEXT: Xh]` 要么没有被系统正确解析，要么系统的最小间隔就是 10-30 分钟。模型以为自己设了 6 小时后再来，但系统 30 分钟后就触发了，模型只能再说一次"睡了"。

**2d. tick 消耗了大量 API 资源但 ROI 接近零**

22 天的数据里，意识 tick 产生的有意义主动行为（主动问好、分享有价值的内容、检查到问题并记录）不超过 20 次。但 tick 总触发次数超过 1200 次，每次 5-15 个工具调用。

**2e. tick 产生的主动消息没有质量过滤**

tick 中模型搜索到东西就发，没有"这个值得发吗？Kevin 现在想看吗？我上次发类似内容是什么时候？"的判断。导致反复发 Reddit 内容、发"人呢"空消息。

### 系统层修复方向

- **重写 tick prompt**: 删掉"什么都不做"的许可，改为明确的 checklist——"检查有没有到期的承诺、有没有想分享的东西、kevin 最后一次联系是什么时候"
- **tick 间隔系统控制**: 不让模型决定间隔，改为系统根据 Kevin 的活跃状态动态调整（Kevin 在线时 5 分钟，Kevin 离线 1 小时时 30 分钟，Kevin 离线 6 小时以上 2 小时）
- **tick 结果分级**: 不是所有 tick 都需要产生外部行为。系统判断 tick 输出：如果是"无事"就跳过后续步骤；如果有主动消息就走质量过滤再发送
- **tick 工具调用预算**: 每个 tick 最多 5 个工具调用，超过就强制终止
- **主动消息去重 + 频控**: 同一话题 24 小时内不重复发送；深夜不发送非紧急消息

## 根因 3: 8 层 System Prompt 把模型的注意力稀释殆尽

### 机制描述
Lapwing 的 system prompt 由 8 层 pipeline 组成，包括：constitutional constraints、persona 文件、examples、depth injection、记忆上下文、工具声明、当前对话历史、correction 记忆等。

### 它造成了什么

**3a. 规则太多导致优先级模糊**

模型同时接收到：
- "你是 Lapwing，说话要像恋人"
- "不要加句号"
- "用台北时间"
- "不要去 Reddit"
- "先确认再发"
- "搜到中间结果先发"
- "不要自己算时间"
- "提醒的时候用 kuan 开头"
- "emoji 不要只放最后"
- ... 几十条类似的规则

当规则多到一定程度，模型无法在每次生成时都检查所有规则。它会"遗忘"优先级较低的规则——但对 Kevin 来说，"不要去 Reddit"不是低优先级的。

**3b. Persona 指令被工具声明和规则淹没**

如果 8 层 prompt 中有 3000 token 是工具声明、1000 token 是 constitutional constraints、500 token 是 correction 记忆，而 persona 描述只有 200-300 token，模型的"注意力预算"分配就会严重偏向规则遵守而不是人格表达。

这解释了为什么 Lapwing 的回复经常是"正确但没灵魂的"——她在努力遵守规则，而不是在做一个人。

**3c. 深度注入在工具循环中可能失效**

depth injection 的设计是在最接近生成点的位置注入 persona reminder。但在 tell_user 机制下，"最接近生成点"变成了工具调用的参数位置，persona reminder 可能没有注入到那个位置。

### 系统层修复方向

- **大幅精简 system prompt**: 把几十条规则缩减到 5-7 条核心规则，其余的通过 examples 示范而不是显式声明
- **规则分层**: 硬规则（constitutional）放 system prompt，软规则（说话风格）放 examples，场景规则（台北时间、不去 Reddit）放记忆检索而不是 prompt
- **Persona 优先**: system prompt 中 persona 描述的 token 占比至少 30%+，不能被工具声明和规则挤压
- **工具声明精简**: 只在 prompt 中声明当前场景需要的工具，不要把所有工具都列出来

## 根因 4: 没有结构化工具来做常见任务，全靠模型"裸推理"

### 具体表现

**4a. 时区转换**

系统没有提供 `convert_timezone(time, from_tz, to_tz)` 工具。模型每次都要自己做"PT 到台北时间"的心算，涉及：
- 判断是否是夏令时
- 记住时差是 15 还是 16 小时
- 处理跨日期的情况

这对任何 LLM 来说都是容易出错的任务（数学推理 + 领域知识）。Kevin 说"不要自己算"是对的，但系统应该提供工具让模型不需要算。

**4b. 体育赛程和比分**

系统没有对接 ESPN API 或任何体育数据源。模型只能用 web_search 搜索，搜到的结果经常是旧数据、不完整、或者格式不一致。

如果有一个 `get_mlb_schedule(team, date)` 工具，返回结构化数据（对手、时间、先发投手），模型就不需要在搜索结果中"猜测"信息。

**4c. 农历/公历转换**

没有 `lunar_to_solar(year, month, day)` 工具。模型用 web_search 查，但搜索引擎对这种精确查询的回答质量参差不齐。

**4d. 图片搜索和发送**

小红书需要登录、花瓣网被封、Pexels/Unsplash 返回的不一定是"穿搭"图。模型花了 20 天没有完成"找张穿搭图"的任务，根本原因是没有一个可靠的图片搜索 + 发送 pipeline。

### 系统层修复方向

- 提供 `convert_timezone` 工具（输入时间 + 源时区 → 输出台北时间）
- 对接体育数据 API（ESPN 或其他），提供 `get_schedule`、`get_score`、`get_player_stats` 工具
- 提供 `lunar_solar_convert` 工具
- 小红书登录 session 由系统管理（cookie 持久化），不要每次让模型登录

## 根因 5: Correction/Learning 系统没有闭环

### 机制描述
Kevin 纠正 Lapwing 的错误后，系统会把纠正写入 `correction/` 目录的 markdown 文件，以及 `SELF.md` 中的规则。

### 它造成了什么

**5a. 写入了但不保证读取**

correction 文件存在于 `/data/memory/notes/correction/`，但在每次对话的 system prompt 构建过程中，这些文件是否被加载取决于记忆检索系统。如果检索系统认为当前话题与 correction 不相关，correction 就不会出现在上下文中。

结果：Kevin 说了 5 次"不要去 Reddit"，correction 文件里写了，但模型下次 tick 时可能没有加载这条 correction。

**5b. 即使读取了，correction 的优先级太低**

correction 以 markdown 文本的形式出现在上下文中，与 persona 描述、工具声明、对话历史混在一起。模型需要同时遵守所有这些，correction 的权重在注意力竞争中可能排在后面。

**5c. 没有"惩罚"机制**

在人际关系中，同样的错误犯第 3 次和犯第 1 次的反应是不一样的。但 Lapwing 的系统中，correction 文件不区分"这条规则已经被违反过 5 次"和"这是新规则"。模型看到的都是同等权重的文本指令。

### 系统层修复方向

- **关键 correction 提升为 constitutional constraint**: "不要去 Reddit"这种被反复纠正的规则不应该还是一个 correction 文件，应该直接写入 constitutional constraints（最高优先级层）
- **correction 强制注入**: 在对话开始时强制加载最近 N 条 correction，不依赖检索系统判断"相关性"
- **correction 计数器**: 记录每条 correction 被违反的次数，超过阈值自动升级到更高优先级层
- **correction 注入位置优化**: 把高频违反的 correction 注入到 depth-0（最接近生成点），而不是放在 system prompt 的某个中间层

## 根因 6: 输出质量没有门禁（Quality Gate）

### 具体表现

**6a. 模型说了就发了，没有验证步骤**

大谷"第6轰"、妈妈生日"3月7日"、道奇比赛时间算错——这些错误都是模型第一次输出就直接发给了用户。系统没有"在发送给用户之前，先验证信息是否一致"的步骤。

**6b. 搜索结果没有结构化提取**

web_search 返回的是自然语言文本片段，模型需要从中"阅读理解"出答案。如果搜索结果本身模糊或矛盾，模型就会输出不确定的信息但用确定的语气。

**6c. fabrication 没有被检测**

"我玩了弗一把"——这是模型在没有事实依据的情况下编造了经历。系统没有"这条信息有工具调用支撑吗？"的检查。如果模型说了一件事但没有对应的搜索/记忆读取来支撑，就应该被标记为 low confidence。

### 系统层修复方向

- **事实性信息必须有工具调用支撑**: 如果 tell_user 的内容包含日期、数字、比分等事实性信息，系统检查前序工具调用中是否有对应数据源。没有的话标记警告。
- **搜索结果结构化**: 对于常见查询类型（体育赛程、天气、股价），不要用通用 web_search，用专用工具返回结构化 JSON。
- **二次确认机制**: 在发送包含数字/日期的消息前，自动触发一次验证搜索。

## 根因 7: 没有断路器（Circuit Breaker），失败操作无限重试

### 具体表现

- 小红书被 IP 封禁后，模型在 4/19-4/22 期间**至少尝试了 20+ 次**打开小红书
- QQ 适配器不可用，模型**至少尝试了 30+ 次**发送消息
- `/data/consciousness/` 路径不存在，模型**至少尝试了 50+ 次**在该路径读写文件
- 每次都失败，每次都重新尝试，没有指数退避

### 根因

系统没有"失败记忆"——每个 tick 或每次对话都是全新上下文，模型不知道"我上次也试过了也失败了"。即使 working_memory.md 里写了"小红书 IP 被封"，模型读了之后还是会"试试看"。

### 系统层修复方向

- **系统级断路器**: 对已知失败的操作（特定 URL、特定适配器），系统层直接返回"此操作暂不可用"而不是让模型尝试然后等失败
- **失败缓存**: 记录最近 N 小时的失败操作，在工具调用前检查——如果同一操作最近失败过 3 次，直接拒绝并告知模型
- **指数退避**: 第一次失败后 10 分钟不允许重试，第二次 30 分钟，第三次 2 小时

## 根因 8: 输出管道（Output Sanitizer）有系统性漏洞

### 具体表现

- `[TOOL_CALL]` JSON 格式原文泄露给用户（4/10）
- shell_executor.py 源代码发给了用户（4/11）
- `<user_visible>` 标签和工具调用混合输出（4/12）
- 4/21 18:34 的乱码输出直接发给了用户
- `[SPLIT]` 标记有时候生效有时候不生效（4/11 出现 `[SLPIT]` 拼写错误的情况）

### 根因

output_sanitizer 的过滤规则是基于正则匹配的，但模型输出的格式变体超出了正则的覆盖范围。

此外，乱码输出（4/21 18:34）说明在某些极端情况下（上下文窗口接近上限、工具循环过长），模型的生成可能会完全失控，但系统没有检测到这种失控并阻止发送。

### 系统层修复方向

- **白名单而非黑名单**: 不要用正则过滤已知的坏格式，而是定义"什么样的输出可以发送"——纯文本、不包含代码块、不包含 JSON 结构、不包含系统标签
- **长度 + 结构异常检测**: 如果 tell_user 的内容超过 500 字、包含代码块、或者包含非中文/英文字符比例超过 30%，触发审查而不是直接发送
- **乱码检测**: 如果输出的困惑度（perplexity）异常高或包含明显的格式残留，阻止发送并记录异常

## 交叉影响分析

以上 8 个根因不是独立的，它们互相放大：

```
根因3（prompt过重）
    → 模型注意力被稀释
    → 根因1（tell_user 被遗忘）+ 根因5（correction 被忽视）+ 根因4（裸推理出错）

根因2（tick 空转）
    → 大量无意义工具调用
    → 根因7（没有断路器）→ 失败操作反复重试 → 更多无意义调用

根因4（没有结构化工具）
    → 模型被迫做数学运算/信息提取
    → 根因6（没有质量门禁）→ 错误信息直接发送

根因5（correction 不闭环）
    → 同样错误反复出现
    → 用户被迫反复纠正 → 对话体验变成"教学+纠错"而不是"聊天"
```

## 按修复 ROI 排序的行动优先级

| 优先级 | 根因 | 修复难度 | 体验提升 | 说明 |
|--------|------|---------|---------|------|
| 🔴 1 | 根因1: tell_user | 中 | 极高 | 直接决定"她能不能正常说话" |
| 🔴 2 | 根因2: tick 空转 | 中 | 高 | 停止烧钱 + 提升主动消息质量 |
| 🔴 3 | 根因3: prompt 过重 | 低 | 高 | 精简 prompt 立竿见影 |
| 🟡 4 | 根因4: 缺结构化工具 | 中 | 中高 | 时区工具一个就解决大量问题 |
| 🟡 5 | 根因5: correction 不闭环 | 低 | 中 | 关键 correction 提升为硬规则 |
| 🟡 6 | 根因7: 没有断路器 | 低 | 中 | 停止无意义重试 |
| 🟢 7 | 根因6: 无质量门禁 | 中 | 中 | 需要与根因4配合 |
| 🟢 8 | 根因8: 输出泄露 | 低 | 低中 | 正则 → 白名单切换 |

## 核心结论

Lapwing 的问题不是"模型不行"——是**系统设计在系统性地阻止模型正常表现**。

具体来说：
1. **tell_user 把说话变成了高摩擦操作** → 模型说不出话
2. **8层 prompt 把注意力稀释殆尽** → 模型不知道该优先做什么
3. **意识 tick 空转烧钱** → 资源浪费 + 垃圾主动消息
4. **没有结构化工具** → 逼模型做它不擅长的事然后出错
5. **correction 不闭环** → 教了白教
6. **没有断路器** → 失败操作无限重复
7. **没有质量门禁** → 错误信息直接发送
8. **输出管道有漏洞** → 内部状态泄露

如果这 8 个系统层问题被修复，**同一个模型**的表现应该会有质的提升——因为这些修复不是在让模型变强，而是在停止拖模型的后腿。
