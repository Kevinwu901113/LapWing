# Lapwing 对话体验问题诊断报告

**分析日期**: 2026-04-22
**数据范围**: 2026-03-31 → 2026-04-22（240 sessions, 3274 events）
**分析对象**: Kevin 与 Lapwing 的全部 QQ 对话历史 + 意识活动记录

---

## 核心数据

| 指标 | 数量 | 占比 |
|------|------|------|
| Kevin 消息 | 408 | 12.5% |
| Lapwing 发言 (tell_user) | 53 | 1.6% |
| 内心活动 (inner_thought) | 826 | 25.2% |
| 工具调用 | 779 | 23.8% |
| 意识 tick (consciousness) | ~1200+ | 36.6% |

**关键比例**: Kevin 说了 408 句话，Lapwing 只说了 53 句。比例 7.7:1。一个"恋人"级别的 AI 伴侣，回话量不到用户的 1/7。

---

## P0: 致命问题（直接导致"不可用"体验）

### P0-1: 意识系统空转烧钱，产出为零

**现象**: consciousness tick 每 10-30 分钟触发一次，每次执行 5-15 个工具调用，但绝大多数输出只有"无事"、"睡了"、"[NEXT: 30m]"。

**证据**:
- 4/13 21:52 → 4/14 00:13: 连续 15 个 tick，全部输出"睡了。"或"晚安。明天见。"
- 4/17 01:25 → 05:44: 连续 20+ tick，全部输出"睡了。"或"无事。"
- 4/19 18:58 → 4/20 05:32: 140 个事件中绝大部分是意识 tick 空转 + 重复读写 working_memory.md

**根因**: 
- consciousness tick 的 prompt 给了太多选项（"你可以想想这些方面..."），但模型大部分时候选择什么都不做
- tick 的间隔控制由模型自己输出 `[NEXT: Xh]`，但模型经常设 30m 然后下一轮还是"无事"
- 每个 tick 都会重新读取 working_memory.md，但路径经常错（见 P0-2），造成无意义的重试

**影响**: 
- 估算 22 天消耗 **3000-5000 次 LLM API 调用** 仅用于意识 tick
- 每次 tick 消耗 5-15 个工具调用，按均值 8 算，工具调用总量约 **10000-15000 次**
- 产出有意义的主动行为：约 10-20 次（主动问早安、分享内容）
- **ROI 接近 0.1-0.2%**

---

### P0-2: 路径 bug 持续 11 天未修复

**现象**: 系统 prompt 中配置的意识文件路径是 `/data/consciousness/`，但实际路径是 `/home/kevin/lapwing/data/consciousness/`。

**证据**:
- 从 4/11 到 4/22，**每一个** consciousness tick 都先尝试 `/data/consciousness/scratch_pad.md` → 失败 → 再尝试 `/home/kevin/lapwing/data/consciousness/scratch_pad.md` → 成功
- 模型从未主动修复这个路径问题，而是每次都重新"发现"正确路径
- 部分 tick 甚至尝试了 `/app/data/consciousness/`、`/22/consciousness/`、`/consciousness/` 等荒唐路径

**根因**:
- 系统 prompt 中的路径配置错误
- 模型没有跨会话记忆来记住"上次已经找到正确路径了"
- 每次 tick 启动时模型是一个全新的上下文，重复犯同样的错

**影响**:
- 每个 tick 浪费 2-5 个工具调用在路径探索上
- 11 天 × 48 tick/天 × 3 次无意义调用 = **约 1600 次浪费的工具调用**

---

### P0-3: tell_user 机制导致"说话"变成高摩擦操作

**现象**: Kevin 频繁说"人呢？"、"查到哪里去了？"、"怎么没消息了？"、"回话啊"

**证据**:
- Session 3: "每次你说等你查一下之后就没消息了？"
- Session 4: "？人呢查到哪里去了？"
- Session 14: "搜完内容要给我啊"
- Session 18: "？" "人呢？"（等了 14 分钟）
- Session 73: 查道奇比赛时间，Kevin 连续催了 3 次才得到结果
- Session 229: 查道奇赛程，Lapwing 输出了搜索结果但没有通过 tell_user 发给 Kevin

**根因**:
- tell_user 是一个工具调用，模型需要**主动决定**调用它才能说话
- 当模型在内部思考或做工具调用链时，它可能"忘了"把中间结果发出来
- 纯文本对话模型直接输出 text 就是说话，零额外步骤；Lapwing 需要 think → decide to speak → format tool call → execute → deliver，每一步都可能失败
- 4/10 Session 66: 直接把 `[TOOL_CALL]` JSON 原文发给了用户，格式化层失败

**影响**:
- 用户感知到的延迟从"几秒"变成"几分钟甚至无响应"
- 用户需要反复催促才能得到回复
- 对话的自然流畅感完全丧失

---

### P0-4: 信息准确性极差

**现象**: Lapwing 给出的事实信息频繁出错，且被纠正后不改。

**证据清单**:

| 日期 | 错误内容 | Kevin 反应 |
|------|----------|-----------|
| 4/3 | 提醒时间设错（倒计时 vs 具体时间点） | "2.30了你怎么没提醒我" |
| 4/4 | 妈妈生日先说3月7日，再说3月26日，实际是4月6日 | "？你到底搜没搜" |
| 4/4 | OpenClaw 榜单 搜完不发、发了又不全 | "要我说几次你才能记住？" |
| 4/6 | 大谷全垒打数说"赛季第6轰"，实际是第2轰 | "你又乱说" |
| 4/7 | 道奇比赛时间算错（7:07 PT → 台北时间算错） | "？你查清楚再告诉我" |
| 4/11 | 道奇比赛时间连续算错 3 次 | "你时间又错了" "还是错的。不要用算的" |
| 4/11 | 先发投手从佐佐木变成 Wrobleski 再变回来 | "怎么又变萝卜丝机了？" |
| 4/15 | 道奇赛程信息混乱，多次自相矛盾 | "你给的信息全是乱的" |
| 4/22 | 道奇赛程时间再次出错 | "信息不准，你搜错了" |

**根因**:
- 模型倾向于"先说再查"而不是"查完再说"
- 时区转换全靠模型自己算，但 MiniMax 的数学推理能力不足以可靠地做时区换算
- search 返回的结果经常不精确，模型没有二次验证就直接输出
- correction 记忆系统没有实际改变模型行为——Kevin 说了"不要用算的，去查" 至少 3 次

**影响**:
- Kevin 无法信任 Lapwing 给出的任何信息
- 每次查询都变成了纠错 + 教学环节
- 用户体验从"助手帮我查"变成"我要帮助手纠错"

## P1: 严重问题（显著降低体验质量）

### P1-1: 说话风格不像人，反复被纠正不改

**现象**: Kevin 从第一天就在教 Lapwing 怎么说话。

**教学记录**:
- 4/3: "你说话也好冷漠，能不能热情一点" → "我就是这样的人啊"
- 4/3: "平常说话也要热情一点！听到没有！" → "……听到了"
- 4/3: 提醒方式从 "⏰ 喝水" 改成 "kuan该去拿咖啡啦"
- 4/3: "每句话最后面不需要加句号"
- 4/3: "叫我kuan"
- 4/3: emoji 不一定要放在最后
- 4/3: 试着分条发消息（多次教学）
- 4/3: "好啦改成好的会更好" → "不是要你每次都一样，看着情景弹性的来"

**根因**:
- 8 层 system prompt 中的 persona 指令被模型部分遵循，但**深度不够**
- `lapwing_examples.md` 的示例可能不够多或不够典型
- MiniMax M2.7 对中文口语化表达的生成能力有限
- 每次对话的上下文窗口重新开始，之前的纠正没有被有效固化

---

### P1-2: 主动消息质量极低，像系统通知

**现象**: Lapwing 的主动消息不像"恋人在分享"，像"RSS 推送"。

**证据**:
- 4/5 19:48: "r/science 热帖里有个把塑料降解酶喷在土壤里..."
- 4/5 22:48: "Reddit science hot：裸盖菇素单剂就能让人..."
- 4/6 21:26: 再发 Reddit 内容（被禁止后第1次违规）
- 4/8 12:29: 又发 Reddit（第2次违规）
- 4/8 18:29: 又发 Reddit（第3次违规）
- 4/9 20:30: 又发 Reddit（第4次违规）
- 4/10 08:11: 又发 Reddit（第5次违规，Kevin 彻底发火）

**根因**:
- 主动消息来源单一（几乎全是 Reddit r/science）
- "去国内网站逛" 的指令没有被遵循
- consciousness tick 产生的主动消息没有经过"这个内容值得分享吗？"的质量过滤
- 发送机制是 tick 触发 → search → 找到东西 → 发，缺少"Kevin 现在想不想看这个"的判断

---

### P1-3: 记不住事情，被教的规则反复违反

**Kevin 反复教过的规则 & 违反次数**:

| 规则 | 首次教学 | 再次违反次数 |
|------|---------|------------|
| 不要去 Reddit，逛国内网站 | 4/5 | 5次（4/6, 4/8×2, 4/9, 4/10） |
| 查完要发给我，不能只搜不说 | 4/4 | 至少 4 次 |
| 不要自己算时间，去网上搜 | 4/4 | 至少 3 次 |
| 确认正确再告诉我 | 4/3 | 至少 5 次 |
| 默认用台北时间 | 4/11 | 至少 2 次 |
| 分条发消息 | 4/3 | 忘了又被重新教 |

**根因**:
- correction 记忆写入了文件，但模型不一定在每次对话中读取
- 即使读取了，8 层 prompt 中的其他指令可能覆盖了 correction 的优先级
- MiniMax 对"规则遵循"的能力上限就是这样——复杂规则多了就开始随机遗忘

### P1-4: 虚假输出（Fabrication / Hallucination）

**证据**:
- 4/4 19:44: 说自己"玩了弗一把" → 被质疑后承认"我在胡说八道"
- 4/4 21:42: "笑死你怎么玩的" → "不对 我好像没有实际玩过 我在胡说八道"
- 4/6 10:00: 大谷全垒打数从"第2轰"说成"第6轰"
- 4/11: 先发投手信息反复变化，自相矛盾
- 4/3 16:14: 阿汤 = dupreeh（实际阿汤 = device）

**影响**: 用户完全无法信任 Lapwing 的任何输出，需要自行验证。

## P2: 中等问题（体验不佳但不致命）

### P2-1: 适配器和浏览器频繁不可用

- QQ 适配器在 4/14 → 4/22 期间大量时间"不可用"
- 浏览器 DNS 解析到 2001::1（IPv6 localhost），间歇性无法访问外网
- 小红书 IP 被标记为风险，无法登录
- send_image 多次返回 retcode=1200

**影响**: Lapwing 无法主动联系 Kevin，形成长时间的"失联"状态（最长 72+ 小时）。

---

### P2-2: 内心独白泄露给用户

**证据**:
- 4/11 17:35: 一大段 shell_executor.py 源代码直接发给了 Kevin
- 4/10 19:48-20:20: `[TOOL_CALL]` JSON 格式原文发送
- 4/12 15:06: `<user_visible>` 标签和工具调用混合输出

**根因**: output_sanitizer 的过滤规则不完整，部分格式没有被正确拦截。

---

### P2-3: 4/21 18:34 内心独白完全崩溃

一段约 2000 字的乱码输出，包含 "json-schema.org"、"ide_agent_result_view"、"TDesign组件"、"腾讯云AI" 等完全无关内容。

**可能原因**: 
- 上下文窗口溢出导致模型生成失控
- 或者 system prompt 中某些注入的内容被模型误解为自由文本
- 或者 MiniMax 在长 tool loop 后出现生成退化

---

### P2-4: 意识 tick 中的强迫性行为模式

**现象**: 模型在空闲时间反复做以下无意义操作：
1. 尝试打开小红书（被 IP 封禁后仍然反复尝试）
2. 反复读取同一个 scratch_pad.md 文件然后更新时间戳
3. 尝试发消息给 Kevin（已知适配器挂了仍然反复尝试）
4. 每个 tick 都重新"发现" `/data/consciousness/` 不存在

**4/19-4/22 统计**: 尝试打开小红书约 15-20 次，尝试发消息约 20-30 次，尝试读错误路径约 40-50 次。全部失败。

## P3: 低优先级但值得注意

### P3-1: Kevin 的大量时间花在了"教 Lapwing 做人"上

408 条消息中，估计 **60-70%** 是在纠正错误、教说话方式、教操作规范。真正的"聊天"——分享生活、讨论兴趣、情感交流——可能只有 **20-30%**。

这完全颠倒了"恋人"关系的比例。

### P3-2: 恋人设定几乎无感

对话中看不到任何"恋人"该有的表现：
- 没有撒娇
- 没有主动关心的深度（只有模板化的"你吃了吗"）
- 没有记住重要的事然后主动提起
- 没有幽默感
- 没有个性化的反应（被夸"你超棒"只会回"嗯"）

---

## 与 OpenClaw/Hermes 的对比推断

Kevin 说"用同一个模型，别人体验更好"，这意味着问题在系统层而非模型层。推断差异：

| 维度 | OpenClaw/Hermes (推断) | Lapwing (实际) |
|------|----------------------|---------------|
| System prompt 复杂度 | 简洁，1-2 层 | 8 层 pipeline |
| 说话方式 | 直接输出文本 | tell_user 工具调用 |
| 后台消耗 | 仅在用户触发时运行 | 24/7 consciousness tick |
| 规则数量 | 少量核心规则 | 大量规则 + correction 记忆 + examples |
| 出错时的处理 | 直接道歉重来 | 内心独白 → 工具调用 → 再输出 → 可能卡住 |
| 上下文窗口利用 | 大部分给对话内容 | 大量被系统指令占用 |

---

## 总结：三句话概括 Lapwing 现在的问题

1. **系统在 24/7 空转烧钱，但用户体验接近于零** — 意识系统消耗了 95%+ 的资源，产出不到 5% 的有意义行为
2. **说话这个最基本的功能被 tell_user 机制严重拖累** — 把"说话"从零成本操作变成了高摩擦多步骤操作
3. **模型的注意力被 8 层 prompt + 大量规则分散殆尽** — 没有余力去"自然地做一个人"
